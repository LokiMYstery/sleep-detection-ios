# 入睡检测 POC 实验路线与验证规划

## 1. 文档目的

本文档用于为后续 SPEC 文档做铺垫，先从实验设计和验证规划的角度，明确本次入睡检测 POC 要验证的路线、统一的评估口径、数据沉淀方式，以及有 Apple Watch 时的联动方案。

本文档不涉及具体代码设计、模块拆分、接口字段和工程实现细节。

当前 POC 也不涉及助眠内容的实际播放与关闭逻辑，但会保留一个“开始当晚记录”的 session 入口，用于模拟真实产品里“开始助眠”的时刻，并为 Watch 侧开始后台采集和回传创造条件。

## 2. POC 总体目标

本次 POC 的目标不是一次性确定唯一正确算法，而是并行验证多条路线，回答以下问题：

1. 在不同设备条件下，哪些信号组合最有机会稳定估计用户真实入睡时间。
2. 无 Watch 与有 Watch 两类场景下，哪条路线最适合作为后续产品主路线。
3. 哪些特征和参数可以在用户已授权 HealthKit 时，于 day0 完成个体化初始化，而不是完全依赖长期在线学习。
4. 如何把每天的数据完整记录下来，供后续调参、回放、误差分析和路线比较。
5. 在用户未授权 HealthKit 的情况下，是否仍能保留一条可运行的 baseline 路线。

## 3. 本次 POC 的统一原则

### 3.1 Ground Truth 定义

本次各路线验证时，优先采用“次日 HealthKit 更新后的睡眠结果”作为离线 ground truth。

当次日确实存在对应夜间睡眠记录时，建议把 HealthKit 中该条记录的入睡开始时间视为统一标签来源。这样做的价值是：

- 各路线共享同一套评估口径，便于横向比较。
- 不要求当天实时知道真值，更符合 POC 阶段的实验逻辑。
- 可以把实时判断和次日标签解耦，先重点验证路线可行性。

需要明确的是，HealthKit 真值更适合作为 POC 阶段的对齐标签，而不是严格医学金标准。因此后续评估时，应把它视为“统一产品标签”，而不是绝对精确的人体生理真值。

如果某一晚次日没有可用的 HealthKit 睡眠记录，则该晚应标记为“无睡眠标签样本”。这类样本仍然可以用于验证数据采集链路、实时判定链路和先验初始化策略，但不能进入依赖 ground truth 的误差评估。

### 3.2 POC 输出物

POC 需要产出两类结果：

- 实时侧结果：某一路线在当晚给出的“预测入睡时间”与判定证据。
- 离线侧结果：次日根据 HealthKit 真值回填后的误差分析、样本归档和参数调整依据。

### 3.3 实验条件原则

本次 POC 需要再明确一个实验组织原则：

- 默认实验条件是设备具备 HealthKit 能力，且具备 Apple Watch 联动能力。
- 在这一套尽可能完整的采集条件下，每晚并行跑多条路线。
- 对无 Watch 路线、无生理先验路线、无某类信号路线的验证，优先通过数据屏蔽或特征禁用来模拟，而不是通过反复切换权限来验证。

这样做的原因是：

- 同一晚的数据条件更一致，路线间更容易直接比较。
- 不需要把“权限差异”与“路线能力差异”混在一起。
- 可以更稳定地复用同一条 session 的原始数据做多路线回放。

### 3.4 评估重点

每条路线都至少回答以下问题：

- 能否稳定给出“预测入睡时间”。
- 对同一个人是否具有可重复性。
- 对摆放方式、是否佩戴 Watch、是否有环境噪音是否敏感。
- 与次日 HealthKit 真值相比，误差主要落在哪个区间。
- 是否具备继续演进为产品能力的潜力。

建议统一观察以下指标：

- 绝对误差中位数。
- 误差分布在 `5 / 10 / 15 / 20` 分钟内的命中率。
- 提前判睡和延后判睡的占比。
- 无法判定、无数据或部分数据样本的比例。

## 4. 我们要验证的实验路线

为了让后续讨论聚焦，建议本次 POC 先明确验证以下五条路线。它们不是互斥关系，而是从最轻量到最重度逐步升级。

这里需要先明确一个划分原则：

- “定时估计”解决的是没有实时传感器证据时，最保守能做到什么。
- “放下手机锚点”解决的是能否利用实时惯性信号识别用户结束用机、准备入睡的时刻。
- “持续体动检测”解决的是手机放在床侧或床面后，能否继续观察体动变化并逼近真实入睡。

因此，路线 A、B、C 虽然都和手机行为或运动有关，但实验问题并不相同，不建议完全合并成一条路线。

另外，本规划不以“可直接读取过去 7 天陀螺仪历史数据”为前提。这里提到的手机惯性信号，默认都指 session 开始后可实时采集的信号；跨天的历史先验仍主要来自 HealthKit 或应用内自行积累的历史样本。

### 路线 A：无 HealthKit 依赖的定时估计基线

这是所有路线的共同 baseline，也应作为兜底方案存在。

核心假设：

- 用户的入睡时间存在一定稳定性。
- 即使没有实时检测，也可以根据手动输入或应用内累积历史估计一个相对可用的入睡时间。

可用输入：

- 用户手动设置的目标入睡时间。
- 应用内最近若干晚记录出的入睡时间或入睡潜伏时间。
- 工作日与周末的差异。
- 当晚 session 开始时间与历史真实入睡时间差值。

可选增强：

- 如果用户授权 HealthKit，可额外引入最近 `7-14` 天睡眠开始时间作为增强，但该能力不应成为路线 A 的前提。

验证重点：

- 在不依赖实时传感器的前提下，误差底线能做到什么程度。
- 它是否足以作为所有高级路线的兜底 baseline。
- 当没有 HealthKit 权限时，它是否仍能作为最低可用版本存在。

从用户视角的输入设计：

路线 A 需要明确成“用户愿意回答什么问题”，而不是“产品想拿什么字段”。

建议优先考虑以下低打扰问题：

- 你通常几点准备睡觉。
- 你一般躺下后，大概多久会睡着。
- 工作日和周末的入睡时间差别大不大。
- 你希望系统更偏保守还是更偏激进地估计入睡时间。

这些问题的设计原则应是：

- 问题少，首次配置尽量控制在 `2-4` 个问题。
- 问题要贴近用户体感，不要求用户理解专业术语。
- 即使用户不授权 HealthKit，也能仅靠这些问题建立 baseline。
- 后续可以用应用内历史自动修正，而不是反复打扰用户。

价值定位：

- 最低成本基线。
- 无 HealthKit、无 Watch、无其他敏感权限时的兜底策略。
- 后续所有复杂路线的对照组。

### 路线 B：无 Watch 的“放下手机锚点”估计

这是非常值得单独验证的一条轻量路线，因为它不要求手机必须持续放在床上感知震动，而是利用用户睡前使用手机行为的结束时刻，配合实时惯性信号来估计入睡。

核心假设：

- 对很多用户来说，“真正放下手机准备睡觉”到“真正入睡”之间存在可学习的时间差。
- 这个“放下手机”时刻可以由交互停止和惯性信号趋稳共同定义。

可用输入：

- session 开始时间。
- 锁屏时间、最后交互时间。
- 屏幕熄灭后的再次唤醒情况。
- 手机实时惯性信号，例如 device motion、陀螺仪、加速度变化。
- 历史上“放下手机时刻”与最终入睡时间的差值分布。

day0 先验来源：

- 最近 `7-14` 天夜间“放下手机时刻”与 HealthKit 入睡时间的差值。
- 工作日与周末分布。
- 最近是否存在“放下手机后很快睡着”或“放下后长期未睡”的模式。

验证重点：

- 仅用“交互结束 + 惯性趋稳”这一轻量信号，不要求手机持续放在床上时，能否给出可用预测。
- 与路线 A 相比，是否明显缩小误差。
- 是否适合作为“无 Watch 主方案”中的轻量分支。

风险：

- 用户可能锁屏后仍长时间清醒。
- 用户可能把手机放远处，导致后续行为不可见。
- 睡前看其他设备时，手机行为与真实入睡脱钩。

与路线 A 的关系：

- 路线 B 可以理解为“路线 A + 一个实时放下手机锚点”。
- 但它仍应独立验证，因为它新增了实时惯性证据，实验价值不同于纯定时估计。

价值定位：

- 如果实时惯性信号的权限和接入成本明显低于麦克风与 HealthKit，则应优先作为无 Watch 轻量路线验证。
- 适合优先验证“低侵入、低权限成本”的上限。

### 路线 C：无 Watch 的 iPhone 体动检测

这是 PDF 中已有的经典轻量检测路线，适合验证“当手机放在床侧或床面后，只靠持续运动信号”能走到什么程度。

核心假设：

- 当手机放在床边、枕边或床垫可感知位置时，设备体动可近似反映用户卧床后的运动变化。

可用输入：

- 加速度 RMS。
- 单位时间峰值次数。
- 姿态变化频率。
- 持续静止时长。
- session 开始后至今的运动变化趋势。

验证重点：

- 在手机摆放位置可控的前提下，是否能比路线 A、B 更接近真实入睡。
- 对“躺着不动但没睡着”的误判比例有多高。
- 不同摆放位置、不同床垫、双人同床时鲁棒性如何。

价值定位：

- 无 Watch 情况下的纯传感器路线。
- 用于判断“手机接触床面”这一前提成立时，体动信号是否值得继续投入。

与路线 B 的边界：

- 路线 B 只需要识别“用户已放下手机”，手机可以在床头柜等非床面位置。
- 路线 C 要求手机持续处在可反映人体体动的位置，目标是做更接近实时检测，而不仅是估计。

### 路线 D：无 Watch 的 iPhone 多模态融合

这是无 Watch 路线里最值得重点验证的一条主线，建议定义为“手机主路线候选”。

核心假设：

- 单独使用体动容易把安静清醒误判为入睡。
- 如果把体动、音频、交互停止、锁屏后状态等信息一起看，能够更接近真实入睡过程。

可用输入：

- 路线 C 的体动特征。
- 麦克风层面的呼吸、翻身、摩擦、环境交互声变化。
- 锁屏后是否仍有交互。
- session 开始后的交互行为变化。
- 历史上该用户的入睡潜伏时间分布。

建议判断方式：

- 先有一个“候选入睡窗口”。
- 在该窗口内同时考察低运动、低交互、音频趋稳三个方向是否共同成立。
- 当多个证据持续满足若干分钟后，才判定为入睡。

验证重点：

- 是否能明显优于纯加速度方案。
- 在环境噪声存在时是否仍能维持可用表现。
- 权限成本、功耗和用户接受度是否在可接受范围内。

价值定位：

- 无 Watch 用户的主推荐实验路线。
- 后续如果要做产品化，这条路线最有可能成为无 Watch 主方案。

### 路线 E：有 Watch 的 Watch 主导融合方案

这是本次 POC 中最关键的一条高价值路线，建议定义为“有 Watch 主路线候选”。

核心假设：

- Watch 腕部体动和心率变化能够显著提升“安静但清醒”和“真正入睡”的区分能力。
- iPhone 仍然提供重要上下文，例如交互停止、是否锁屏、手机侧惯性变化、可选环境音等。

可用输入：

- Watch 侧腕部体动。
- Watch 侧心率绝对值、下降幅度、变化斜率。
- iPhone 侧 session 状态和交互结束时间。
- iPhone 侧可选环境上下文。
- 历史 HealthKit 睡眠样本与心率样本。

day0 先验来源：

- 最近若干晚的睡前心率基线。
- 历史夜间入睡前后心率变化区间。
- 用户典型入睡潜伏时间。
- 典型“开始当晚记录到心率进入低区间”的时间差。

验证重点：

- 是否显著优于所有无 Watch 路线。
- Watch 心率下降与腕部低动是否足够稳定，可形成明确证据链。
- 如果 Watch 临时断连、未佩戴、数据缺失，样本是否仍能被清晰记录和分析。

价值定位：

- 有 Watch 用户的主推荐实验路线。
- 最有机会得到高置信度入睡判定的方向。

## 5. 推荐的路线优先级

为了便于后续一条路线一条路线深入讨论，建议优先级如下：

1. 路线 E：有 Watch 的 Watch 主导融合方案。
2. 路线 B：无 Watch 的“放下手机锚点”估计。
3. 路线 D：无 Watch 的 iPhone 多模态融合。
4. 路线 C：无 Watch 的 iPhone 体动检测。
5. 路线 A：无 HealthKit 依赖的定时估计基线。

推荐这样排序的原因是：

- 路线 E 最能体现差异化价值，也最需要先把端间协同想清楚。
- 路线 B 在无 Watch 场景里权限和侵入性更轻，适合优先验证其上限。
- 路线 D 是无 Watch 场景最像正式产品主线的路线，但因为涉及麦克风，更适合作为第二阶段重点验证。
- 路线 C 更像对单一传感器上限的测量。
- 路线 A 虽然简单，但作为 baseline 和兜底必须存在。

## 6. 有 Watch 时的联动方案

这一部分不讨论代码接口，只定义职责边界和实验流程。

### 6.1 关键待验证问题

有 Watch 路线里，本轮 POC 需要尽早验证一个关键问题：

`为了拿到足够实时的 Watch 侧数据，我们是否必须编写并启动 Watch app。`

这会直接影响后续路线 E 的工程边界和实验组织方式。

当前建议不要在文档里预设答案，而是把它列为显式验证项：

- 仅依赖 iPhone 侧读取 HealthKit，同步延迟是否足够小。
- 是否只有在 Watch 侧存在 app 且主动启动 session 时，才能拿到足够连续、足够实时的腕动与心率信号。
- 如果需要 Watch app，它是否还需要配合某种持续运行机制，才能满足夜间记录需求。

### 6.2 角色分工

iPhone 负责：

- 提供“开始当晚记录”的 session 入口，可用于模拟真实产品中的助眠开始时刻。
- 记录用户交互、锁屏、前台后台状态。
- 发起和管理当晚实验 session。
- 聚合多端特征、生成当晚预测结果。
- 本地持久化和导出实验数据。
- 次日拉取 HealthKit 真值并回填标签。

Watch 负责：

- 在 session 期间持续提供腕部运动相关信号。
- 提供心率及其趋势变化。
- 在需要时回传简化后的特征窗口或阶段性状态。
- 在 iPhone 不可达时尽量保留关键时间段摘要，待恢复连接后补传。

HealthKit 负责：

- 作为跨天的统一历史样本来源。
- 提供次日 ground truth。
- 为 day0 个体化参数初始化提供历史先验。

### 6.3 推荐的联动逻辑

建议以 `iPhone 为 session 主控，Watch 为实时传感主源`。

原因是：

- session 入口创建和多端状态协调更适合由 iPhone 管理。
- 数据归档、导出和次日真值回填更适合统一由 iPhone 管理。
- Watch 更适合承担贴近人体的实时采集职责，而不适合承担实验主状态机。

### 6.4 当晚 session 的理想流程

1. 用户在 iPhone 侧点击“开始当晚记录”。
2. iPhone 创建一条当晚实验 session，并记录开始时间、路线类型和设备可用性。
3. 如果检测到可用 Watch，则进入 Watch 联动模式，同时下发当晚 session 上下文。
4. Watch 从 session 开始后持续产出腕动和心率相关特征。
5. iPhone 同时记录锁屏、交互停止、可选环境信息和手机侧惯性变化。
6. iPhone 基于双端证据进入“清醒”“疑似入睡”“确认入睡”这类实验状态。
7. 当满足判定条件时，iPhone 记录预测入睡时间和对应证据链。
8. 次日 HealthKit 睡眠结果更新后，iPhone 把该 session 回填为已标注样本。

### 6.5 推荐的判定思路

有 Watch 时建议使用“证据叠加”而不是单阈值。

至少包含三类证据：

- Watch 证据：腕部低动持续存在。
- Watch 证据：心率相对睡前基线下降到某一区间，或表现出持续下降趋势。
- iPhone 证据：交互结束，且手机侧行为与惯性信号趋于稳定，没有明显清醒行为。

当三类证据中的两到三类在一段连续时间内共同成立时，可进入“疑似入睡”；若继续保持，则进入“确认入睡”。

### 6.6 本轮 POC 的边界

本轮 POC 暂不设计复杂回退逻辑。

当前只需要明确以下处理方式：

- 如果当晚没有 Watch 相关数据，则直接记录为“无 Watch 数据样本”。
- 如果 Watch 只有部分数据，则记录为“部分 Watch 数据样本”。
- 不要求在本轮 POC 中自动切换到其他路线继续判断。

这样可以保证：

- 数据标注口径清楚。
- 后续分析时能够区分“完整 Watch 样本”“部分 Watch 样本”和“无 Watch 数据样本”。

## 7. 数据记录与导出规划

POC 阶段必须把“实时判断”变成“可回看、可分析、可导出”的样本库，否则后续无法系统调参。

### 7.1 每晚至少要记录的内容

建议每条 session 至少记录以下几类数据：

- 基本信息：日期、开始时间、结束时间、路线类型、是否有 Watch、Watch 数据完整度。
- 上下文信息：是否锁屏、最后交互时间、是否有麦克风权限、手机摆放方式、session 入口触发方式。
- 特征摘要：按固定时间窗记录体动、音频、心率、交互停止时长等摘要特征。
- 判定过程：何时进入候选状态、何时进入疑似入睡、何时确认入睡、为何触发。
- 结果标签：当晚预测入睡时间、次日是否存在 HealthKit 睡眠标签、如存在则记录真值与误差。
- 补充标签：是否为工作日、是否异常晚睡、是否中途醒来、是否手动干预、Watch 数据是否完整。

### 7.2 导出形式

建议至少支持两层导出：

- `session 级导出`：一晚一条汇总记录，便于看总体误差和路线对比。
- `window 级导出`：固定时间窗的特征序列，便于之后调阈值、回放和做离线分析。

导出格式建议保持简单：

- 首选 `CSV + JSON`。
- CSV 适合快速看汇总统计。
- JSON 适合保留时间序列和判定过程。

### 7.3 为什么导出很重要

如果只有“是否判定成功”这种结果，后续很难知道问题出在：

- 特征本身没信号。
- 阈值不合理。
- 个体差异太大。
- 端间协同丢数据。
- Ground truth 对齐有偏差。

因此，POC 的关键不是只做一次实时判断，而是构建一个可用于反复校准的实验样本池。

## 8. Day0 与历史先验规划

本次 POC 很值得单独验证的一点，不是“默认存在 HealthKit 睡眠先验”，而是“先判断用户当前实际拥有哪类先验，再决定第一晚能初始化什么参数”。

这部分尤其重要，因为在无 Watch 场景下，我们很可能拿不到任何睡眠相关的 HealthKit 历史；即使用户授权了 HealthKit，也未必真的有睡眠样本。

### 8.1 本部分的核心原则

对每个用户，POC 在 day0 都应先做一次“先验可用性探测”，而不是直接假设有完整历史。

建议按以下顺序探测：

1. 是否存在 HealthKit 睡眠样本。
2. 是否存在可用的生理先验，例如心率、HRV、静息心率等。
3. 如果以上都没有，是否只能依赖手机侧实时惯性信号和后续逐日积累的应用内历史。

这意味着，day0 并不是一定能初始化所有路线，而是要先把用户归类到不同的“先验可用层级”。

### 8.2 建议的先验可用层级

建议把用户分成三类：

- `P1：有睡眠先验`
  存在可用的 HealthKit 睡眠样本，可直接用于入睡时间、入睡潜伏期等初始化。
- `P2：无睡眠先验，但有生理先验`
  没有睡眠样本，但存在心率、HRV、静息心率等数据，可用于有 Watch 路线的阈值初始化。
- `P3：无可用 HealthKit 先验`
  没有可用睡眠样本，也没有足够生理先验，此时只能依赖路线 A 的手动输入，以及路线 B/C/D 在 session 期间采集到的实时信号，并从 D1 开始积累应用内历史。

这样划分的好处是：

- 不会把“授权了 HealthKit”误认为“就一定有足够先验”。
- 能清楚地区分无 Watch 用户和有 Watch 用户的初始化能力差异。
- 后续分析时可以比较 `P1 / P2 / P3` 三类用户的误差差异。

### 8.3 无 Watch 场景下的先验策略

无 Watch 场景需要明确一个现实前提：

- 很多用户可能没有任何可用的睡眠类 HealthKit 历史。
- 因此，无 Watch 路线不能把睡眠先验当作默认前提。

在这种前提下：

- 如果用户属于 `P1`，可以使用睡眠样本去初始化入睡时间分布、工作日/周末差异、入睡潜伏期，以及“开始当晚记录”到入睡的时间差。
- 如果用户属于 `P2` 或 `P3`，路线 A 只能依赖用户输入和应用内历史。
- 如果用户属于 `P2` 或 `P3`，路线 B/C/D 的第一晚主要依赖 session 开始后的实时惯性信号与行为上下文，不应宣称存在真正的跨天先验。

也就是说，无 Watch 场景下所谓的“先验”很可能不是 HealthKit 提供的，而是：

- 用户主动输入。
- session 期间实时识别出的“放下手机锚点”。
- 从 D1 开始逐天积累的应用内历史。

### 8.4 有 Watch 场景下的先验策略

有 Watch 场景下，HealthKit 历史更有可能提供可用生理先验，但仍然不应默认一定存在睡眠样本。

如果用户属于 `P1`，可重点生成以下先验：

- 用户典型入睡时间分布。
- 工作日与周末的入睡差异。
- 入睡前后常见心率变化区间。
- 从“开始当晚记录”到真实入睡的典型时间差。

如果用户属于 `P2`，可重点生成以下先验：

- 睡前一段时间的常见心率基线区间。
- 历史夜间时段的心率波动范围。
- HRV 或静息心率是否能提供个体化边界。

这些先验主要帮助路线 E：

- 避免对所有用户使用统一心率阈值。
- 把“疑似入睡”的判断边界设置得更贴近个体。
- 降低有 Watch 路线的冷启动成本。

### 8.5 Day0 先验的真实目标

需要强调，day0 的目标不是“无论什么用户都能在第一晚做强个性化”，而是验证以下命题：

`当用户具备不同层级的历史先验时，我们能否稳定识别这些先验，并把它们转化为合理的初始参数。`

因此，第 8 部分真正要验证的不是单一算法效果，而是“先验探测与参数初始化策略”本身。

## 9. POC 集成与并行实验方式

本次 POC 不应拆成多个彼此独立的小 demo，而应在一个 POC 中一次集成所有测试路线，并在每天的同一条 session 上并行产出多条路线的实验结果。

这里还需要明确：

- 默认采集条件按“有 HealthKit、可联动 Watch”的完整环境设计。
- 路线 A/B/C/D 的验证不是通过真的去掉权限，而是通过对同一晚 session 做数据屏蔽来模拟。
- 例如验证无 Watch 路线时，可以直接屏蔽 Watch 侧特征；验证无睡眠先验时，可以直接屏蔽睡眠类先验；验证无生理先验时，可以进一步屏蔽心率、HRV 等初始化信息。

### 9.1 并行实验的基本方式

建议每晚只有一条统一的 session，但在这条 session 内并行计算多条路线的结果：

- 路线 A 产出一条 baseline 预测。
- 路线 B 产出一条“放下手机锚点”预测。
- 路线 C 在摆放条件满足时产出一条体动检测结果。
- 路线 D 在权限和环境满足时产出一条多模态结果。
- 路线 E 在 Watch 数据可用时产出一条 Watch 主导结果。

这样做的价值是：

- 同一晚、同一用户、同一 ground truth 下可以直接横向比较。
- 不会因为不同路线在不同夜晚运行而引入额外噪音。
- 每天都会持续积累一批可比较样本，而不是分散到多个阶段性 demo。

### 9.2 并行实验下的数据记录要求

既然是一晚并行跑多条路线，那么每晚记录不仅要保存原始上下文，还要保存“每条路线各自的输出”。

建议每条 session 额外记录：

- 哪些路线当晚被启用。
- 哪些路线因为权限、设备条件或数据缺失未能产出结果。
- 每条路线各自的预测入睡时间。
- 每条路线各自的证据摘要。
- 每条路线相对次日 ground truth 的误差。

### 9.3 为什么第 9 部分要这样定义

如果每次只验证一条路线，会带来两个问题：

- 很难排除“今晚状态特殊”这种夜间噪音。
- 后续路线间比较时，样本条件不一致。

因此，本次 POC 更适合做成“一个统一采集框架 + 多条路线并行出分”的形态。
